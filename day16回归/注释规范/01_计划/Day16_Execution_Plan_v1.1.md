## 0. 本版核心决策

1. 当前阶段优先级调整为：**全项目注释规范化 > Day16 新功能继续开发**。
2. Day16 功能计划不废弃，但进入“冻结执行窗口”，先完成注释专项再恢复功能开发。
3. 注释专项是全项目范围，不局限于 Day16 相关代码。
4. 注释专项执行时**不改业务逻辑**，仅做可读性与规范化改造。

---

## 1. 专项目标（必须达成）

1. 让开发者快速理解代码职责：看方法注释即可判断“做什么、为什么、关键限制”。
2. 让关键业务点可追踪：状态流转、权限校验、幂等、副作用语句必须有注释。
3. 让术语口径统一：`用户 ID / 商品 ID / 订单 ID / 创建时间 / 更新时间 / 分页查询`。
4. 让质量可量化验收：建立扫描门禁并固化到后续评审流程。

---

## 2. 专项范围与排除

### 2.1 范围
1. 代码：`demo-service`、`demo-common`、`demo-pojo` 的 `*.java`。
2. 映射与文档：`*.xml`、`*.md`、`*.sql`、`*.yml`、`*.yaml`（仅注释文本）。
3. 回归文档目录：`day7回归` ~ `day16回归`。

### 2.2 排除
1. `node_modules/`
2. `.idea/`
3. `target/`
4. 数据库备份文件（如 `*dump.sql`）

---

## 3. 完成定义（DoD）

### 3.1 硬门禁（必须 100% 达成）
1. 乱码命中数 = 0（源码与注释文本）。
2. Java 文件 `ZERO_COMMENT = 0`。
3. Java 文件注释密度 `<10%` 的文件数 = 0。
4. `public` 方法 JavaDoc 覆盖率 = 100%（Controller / Service / ServiceImpl）。
5. 术语漂移命中数 = 0（ID 与时间口径统一）。

### 3.2 质量目标（建议）
1. `serviceimpl` 注释密度建议 >= 15%。
2. 状态流转方法至少包含 3 处步骤注释。
3. 所有涉及外部副作用的方法明确说明影响范围（DB/MQ/Redis/Mongo）。

---

## 4. 执行总策略（防走偏）

1. 先“全量扫描建账”，再“分批修复”，最后“全量复检关账”。
2. 每批只做注释相关改动，不混入业务逻辑改动。
3. 每批完成必须通过 4 个检查：
   - 乱码检查
   - 术语漂移检查
   - 注释密度检查
   - 抽样人工复核（关键流程）
4. 出现以下任一情况必须暂停并复盘：
   - 注释与代码行为不一致
   - 语义注释误导业务理解
   - 扫描结果出现回退

## 5. 分批计划（详细）

## Phase 0：基线冻结与任务建账
### 目标
1. 建立全项目注释问题清单（文件、问题类型、优先级）。
2. 冻结专项规则，避免边做边改口径。

### 动作
1. 生成“文件级问题清单”：
   - 无注释
   - 低密度
   - public 方法缺 JavaDoc
   - 术语漂移
   - 历史乱码注释
2. 将问题按优先级分三层：
   - P0：Controller / ServiceImpl / Mapper XML
   - P1：Entity / DTO / VO / Enum
   - P2：文档与脚本注释

### 交付
1. 问题清单（可复用扫描脚本结果）。
2. 分批待办列表。

### 验收
1. 能明确回答“还有哪些文件未达标”。

---

## Phase 1：接口层与核心业务层（P0）
### 目标
1. 先保障“最常读到的代码”可读性。

### 范围
1. `controller/**`
2. `service/**`
3. `service/serviceimpl/**`

### 规范要求
1. 每个 `public` 方法有 JavaDoc。
2. 方法内部关键流程有步骤注释（建议 1/2/3）。
3. 状态流转语句写明 `A -> B`。
4. 权限与幂等分支写明“为什么这样判定”。

### 验收
1. `public` 方法 JavaDoc 缺失数 = 0（上述范围）。
2. 抽样 10 个关键方法，均能在 2 分钟内看懂主流程。

---

## Phase 2：数据访问层与异步链路（P0）
### 目标
1. 让 SQL 口径、异步链路、幂等逻辑可追踪。

### 范围
1. `mapper/*.java`
2. `resources/mapper/*.xml`
3. `mq/**`、`job/**`、`outbox` 相关服务

### 规范要求
1. Mapper 方法注明用途与关键筛选口径。
2. XML 注释说明默认筛选规则与状态口径。
3. MQ/Job 代码注明触发条件、幂等策略、失败补偿。

### 验收
1. 关键 SQL 方法有语义注释。
2. 消息消费链路关键节点（消费/重试/去重）注释完整。

---

## Phase 3：模型层（P1）
### 目标
1. 建立统一字段语义，避免同名字段理解偏差。

### 范围
1. `entity/**`
2. `dto/**`
3. `vo/**`
4. `enumeration/**`

### 规范要求
1. 关键字段要有中文语义注释。
2. 术语统一：`用户 ID / 商品 ID / 订单 ID / 创建时间 / 更新时间`。
3. 枚举注释说明业务语义，不只写字面值。

### 验收
1. 字段术语漂移命中数 = 0。
2. 时间字段注释口径统一。

---

## Phase 4：文档与脚本注释（P2）
### 目标
1. 保证工程文档和 SQL 脚本口径与代码一致。

### 范围
1. `*.md`
2. `*.sql`（仅注释）
3. `*.yml/*.yaml`（仅注释）
4. `*.xml`（注释）

### 规范要求
1. 术语统一，不出现混写。
2. 不修改脚本逻辑，仅改注释文本。

### 验收
1. 文档术语漂移命中数 = 0。
2. 注释乱码命中数 = 0。

---

## Phase 5：全量复检与准入固化
### 目标
1. 从“一次性整改”变成“持续不回退”。

### 动作
1. 全量扫描并输出最终报告。
2. 形成 PR 检查清单并纳入评审模板：
   - public 方法 JavaDoc
   - 状态流转注释
   - 术语统一
   - 乱码零容忍
3. 约定后续新增代码默认按规范执行。

### 验收
1. 达成第 3 章全部硬门禁。
2. 形成可复用检查流程（人工或脚本）。
