## 0. 目标与边界

### 0.1 目标
1. 让任何开发者在 30 秒内看懂方法用途，在 2 分钟内看懂核心实现路径。
2. 对关键业务语句（状态流转、权限校验、幂等、外部副作用）做到“有注释可追踪”。
3. 统一术语，避免同义词和大小写口径漂移。
4. 建立可执行验收标准，防止后续代码回退到“低可读性”。

### 0.2 范围
1. `*.java`：类、接口、方法、关键语句注释。
2. `*.xml`：仅注释块（`<!-- -->`）术语与语义规范。
3. `*.md`：设计与回归文档术语规范。
4. `*.sql`、`*.yml`、`*.yaml`：注释文本规范。

### 0.3 不在本次范围
1. 改动业务逻辑。
2. 重构接口契约。
3. 自动生成 API 文档平台建设（仅约束注释格式）。

## 1. 术语冻结（必须统一）

### 1.1 主术语
1. `用户 ID`（禁止：用户ID、用户Id、用户id）
2. `商品 ID`（禁止：商品ID、商品Id、商品id）
3. `订单 ID`（禁止：订单ID、订单Id、订单id）
4. `地址 ID`、`收货地址 ID`
5. `创建时间`、`更新时间`
6. `分页查询`
7. `状态流转`
8. `幂等`
9. `权限校验`
10. `业务异常`

### 1.2 状态术语
1. 商品状态统一使用：`under_review / on_sale / off_shelf / sold`
2. 注释描述统一写法：`状态流转：A -> B（原因）`

## 2. 注释分层标准（强制）

### 2.1 类级注释（Class JavaDoc）
1. 说明该类的业务职责。
2. 说明该类处理的核心边界。
3. 如有副作用，说明依赖组件（DB/MQ/Redis/Mongo 等）。

### 2.2 方法级注释（Method JavaDoc）
1. 所有 `public` 方法必须有 JavaDoc。
2. 必须包含以下信息：
   - 做什么（业务目标）
   - 关键约束（权限、状态、参数边界）
   - 返回语义
   - 异常语义（抛出条件）
3. 接口层方法（Controller/Service）必须额外说明“入口角色”。

### 2.3 语句级注释（Inline Comment）
1. 对以下语句必须加注释：
   - 状态流转语句
   - 权限判定语句
   - 幂等处理语句
   - 事务关键写操作
   - 外部副作用（MQ、Redis、Mongo、第三方）
2. 注释必须解释“为什么”，不是重复“做了什么语法动作”。

### 2.4 禁止项
1. 禁止乱码注释。
2. 禁止中英文混乱缩写（如 `用户id`、`productID`）。
3. 禁止“废话注释”（如：`// 给变量赋值`）。
4. 禁止过期注释与代码不一致。

---

## 3. 模块级最低要求

### 3.1 Controller
1. 每个接口方法必须有 JavaDoc，说明：
   - 接口作用
   - 调用角色
   - 关键参数
   - 结果语义
2. 关键日志点（入参、主体 ID、结果）要有注释说明用途。

### 3.2 Service / ServiceImpl
1. 每个 `public` 方法必须有 JavaDoc。
2. 方法内部按步骤注释（建议 `1/2/3`）。
3. 状态流转必须写 `A -> B` 注释。
4. 幂等/风控/权限语句必须解释判定原因。

### 3.3 Mapper / XML
1. 每个 SQL 方法要有用途注释。
2. 条件更新 SQL 要注明并发与幂等语义。
3. XML 注释必须说明筛选口径，不写“无信息”注释。

### 3.4 Entity / DTO / VO
1. 关键字段必须有中文语义注释。
2. 时间字段统一：`创建时间`、`更新时间`。
3. 关联字段统一：`用户 ID`、`商品 ID`、`订单 ID`。

### 3.5 Job / MQ / Outbox
1. 注明触发条件、幂等策略、失败补偿方式。
2. 注明副作用范围（写库、发消息、重试）。

### 3.6 Test
1. 测试类注释说明覆盖的业务规则。
2. 测试方法注释说明“Given/When/Then”要点。

## 4. 注释模板（执行时直接套用）

### 4.1 类模板
```java
/**
 * XXX 服务实现。
 * 负责 A、B、C 三类业务，并在 D 场景下执行 E 副作用。
 */
```

### 4.2 方法模板
```java
/**
 * 执行 XXX 操作。
 * 关键规则：先做权限校验，再做状态校验，最后执行条件更新。
 *
 * @param userId 当前用户 ID
 * @param request 请求参数
 * @return 处理结果
 * @throws BusinessException 当权限不足或状态不满足时抛出
 */
```

### 4.3 状态流转语句模板
```java
// 状态流转：under_review -> on_sale（管理员审核通过）
```

### 4.4 幂等语句模板
```java
// 幂等处理：重复请求命中唯一键时返回已成功结果，不重复写库
```

### 4.5 XML 注释模板
```xml
<!-- 管理端分页查询待审核商品。默认口径：未传 status 时只查 under_review。 -->
```

## 5. 质量门禁（验收标准）

### 5.1 硬性门禁
1. 乱码命中数 = 0（源码与文档注释）。
2. Java 文件 `ZERO_COMMENT = 0`。
3. Java 文件注释密度：`<10%` 文件数 = 0。
4. `public` 方法 JavaDoc 覆盖率 = 100%（Controller / Service / ServiceImpl）。
5. 术语漂移命中数 = 0（ID/时间/分页查询口径）。

### 5.2 质量目标（建议）
1. 核心业务类（`serviceimpl`）注释密度建议 `>=15%`。
2. 状态机相关类建议 `>=20%`。
3. 每个关键流程方法至少 3 处步骤注释。

## 6. 执行计划（防走偏版本）

### Step 1：基线扫描（当天）
1. 全量扫描乱码、术语漂移、低注释密度文件。
2. 产出“问题清单 + 文件优先级”。

### Step 2：高风险先补（当天）
1. 优先处理 `controller`、`serviceimpl`、`mapper/xml`。
2. 先补类注释和 public 方法注释，再补关键语句注释。

### Step 3：全量补齐（D+1）
1. 清理 entity/dto/vo/enum 的字段注释口径。
2. 清理 md/sql/yml/xml 注释口径。

### Step 4：回归审查（D+1）
1. 按门禁标准重新扫描。
2. 抽样人工复核关键流程（登录、商品状态流转、订单流程、消息通知）。

### Step 5：固化机制（D+1）
1. 把本规范加入 PR 检查清单。
2. 新增代码评审必查项，避免后续反复返工。

## 7. PR 检查清单（提交前必过）

1. 是否新增/修改了 `public` 方法但未写 JavaDoc？
2. 是否新增状态流转但未写 `A -> B` 注释？
3. 是否新增权限/幂等逻辑但未解释原因？
4. 是否出现术语漂移（用户ID/商品ID/订单ID）？
5. 是否出现乱码或历史乱码字符？
6. 注释是否与代码行为一致？
7. 是否只写“做了什么”而没写“为什么”？
