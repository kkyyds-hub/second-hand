## 2. 接口清单

| 场景 | 接口 / 契约 | 当前优先级 | 文档口径 | 备注 |
|---|---|---|---|---|
| 账户资料编辑 | `PATCH /user/me/profile` | 第一刀 | 作为 Day02 首个最小切片主入口，先承接昵称 / 简介编辑与成功后的 session 回写。 | `UpdateProfileRequest` 当前包含 `nickname / avatar / bio`；首切片不强行同时做头像上传。 |
| 头像上传配置 | `POST /user/me/upload-config` | 后续子流 | 作为头像上传前的配置 / ticket 获取入口。 | 与实际上传组成两步链路，不在第一刀里直接宣称打通。 |
| 头像实际上传 | `PUT /user/me/avatar/upload` | 后续子流 | 作为头像上传第二步提交入口。 | 请求体、文件流与成功回写策略待执行时再补字段级口径。 |
| 密码修改 | `POST /user/me/password` | 后续子流 | 作为安全中心主入口之一。 | `verifyChannel`、验证码 / 原密码校验口径需执行时细化。 |
| 手机绑定 | `POST /user/me/bindings/phone` | 后续子流 | 绑定新手机号，成功后需明确 session 回写与提示语。 | 不能与 Day01 手机注册混为同一条链路。 |
| 手机解绑 | `DELETE /user/me/bindings/phone` | 后续子流 | 解除当前手机号绑定。 | 需要明确验证方式与失败提示。 |
| 邮箱绑定 | `POST /user/me/bindings/email` | 后续子流 | 绑定新邮箱，成功后需同步账户展示。 | 不能直接复用注册期文案。 |
| 邮箱解绑 | `DELETE /user/me/bindings/email` | 后续子流 | 解除当前邮箱绑定。 | 需要记录解绑前置验证口径。 |
| 地址列表 | `GET /user/addresses` | 后续子流 | 作为地址管理页默认入口。 | 默认分页参数是 `page=1`、`pageSize=20`。 |
| 默认地址 | `GET /user/addresses/default` | 后续子流 | 作为默认地址卡片或后续下单前回显入口。 | Day02 只规划入口，不提前承接订单流。 |
| 地址详情 | `GET /user/addresses/{addressId}` | 后续子流 | 作为编辑前回填或详情查看入口。 | 需要明确路由进入方式。 |
| 地址新增 | `POST /user/addresses` | 后续子流 | 作为新增地址表单提交入口。 | 表单字段以 `AddressDTO` 为准。 |
| 地址编辑 | `PUT /user/addresses/{addressId}` | 后续子流 | 作为地址编辑提交入口。 | 与新增共用表单时必须保留提交路径区分。 |
| 地址删除 | `DELETE /user/addresses/{addressId}` | 后续子流 | 作为地址删除操作入口。 | 需要保留删除确认与失败提示策略。 |
| 设为默认地址 | `PUT /user/addresses/{addressId}/default` | 后续子流 | 作为地址列表内显式操作入口。 | 需与默认地址展示保持同步。 |

## 3. Day02 接口对齐原则

1. Day02 继续沿用 Day01 的 `authentication` 请求头、401 清理与 session 基线，不重写认证协议；
2. 账户资料、安全中心、地址管理至少拆成独立 API 模块，不允许页面直接手写接口 URL；
3. 首个最小切片先只对齐 `PATCH /user/me/profile`，并明确成功后通过既有 session helper 回写本地账户快照；
4. 头像上传按“两步链路”冻结：先拿上传配置，再执行真实上传；若执行阶段发现协议差异，必须写回 `05_进度回填`；
5. 任何字段级不确定项都必须写成 `待执行时确认`，不能补脑写成既定事实。
