| 密码修改 | `POST /user/me/password` | 作为安全中心主入口之一。 | `verifyChannel`、验证码 / 原密码校验口径需在执行时细化。 |

| 默认地址 | `GET /user/addresses/default` | 作为下单前地址回显与账户域默认地址卡片入口。 | Day02 只规划入口，不提前承接订单流。 |

| 地址新增 | `POST /user/addresses` | 作为新增地址表单提交入口。 | 表单字段以 `AddressDTO` 为准，待执行时补字段表。 |
| 地址编辑 | `PUT /user/addresses/{addressId}` | 作为地址编辑提交入口。 | 与新增共用表单时，必须保留区分逻辑。 |
| 地址删除 | `DELETE /user/addresses/{addressId}` | 作为地址删除操作入口。 | 需要保留删除确认与失败提示策略。 |
| 设为默认地址 | `PUT /user/addresses/{addressId}/default` | 作为地址列表内显式操作入口。 | 需与默认地址展示保持同步。 |

## 3. Day02 接口对齐原则

1. Day02 延用 Day01 的 `authentication` 请求头、401 清理与 session 基线，不重写认证协议。
2. 账户资料、安全中心、地址管理至少拆成独立 API 模块，不允许页面直接手写接口 URL。
3. 头像上传按“两步链路”冻结：先拿上传配置，再执行真实上传；若执行阶段发现协议差异，必须写回 `05_进度回填`。
4. 账户资料初始展示仍以 Day01 的本地 session 为起点；在没有确认新的读取策略前，不虚构 `GET /user/me`。
5. 任何字段级不确定项都必须写成 `待执行时确认`，不能补脑写成既定事实。
