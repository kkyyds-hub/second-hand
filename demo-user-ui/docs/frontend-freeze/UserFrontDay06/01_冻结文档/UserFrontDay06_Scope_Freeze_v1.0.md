---

## 1. 起点结论

- 结论：`UserFrontDay06 应作为 卖家订单、发货、物流、售后处理与订单会话 的正式入口。`
- 判定依据：
  1. Day05 已把买家订单创建、支付、取消、确认收货与售后发起单独建档，卖家履约必须在此基础上继续拆分。
  2. 后端已存在 `OrdersController`、`AfterSaleController`、`MessageController`，而用户端还没有卖家订单页、物流页与订单会话入口。
  3. 若不独立建档，发货、物流、卖家售后处理和订单会话消息很容易被买家订单日或 Day08 共享治理日稀释。

---

## 2. 做什么

1. 冻结卖家订单列表 / 详情 / 发货 / 物流查看 / 卖家售后处理 / 订单会话消息的契约、API 模块、路由与回填口径。

---

## 3. 不做什么

1. 不回写 Day05 的买家支付与取消
2. 不把系统通知、钱包积分、全局错误治理提前写进 Day06
3. 不把 MessageController 直接写成完整消息中心已交付。

---

## 4. 范围拆解

| 范围分类 | 归属内容 | 处理方式 |
|---|---|---|
| 卖家订单履约 | `OrdersController` seller 链路 | 作为 Day06 主链。 |
| 卖家售后处理 | `AfterSaleController` seller decision | 与 Day05 buyer after-sales 分开。 |
| 订单会话消息 | `MessageController` order chat | 系统通知留给 Day08。 |
| 沿用基线 | Day01 与 Day05 结论 | 只做接棒说明。 |
| 非 Day06 范围 | 系统通知、共享治理、资产域 | 留给 Day07 以后处理。 |
