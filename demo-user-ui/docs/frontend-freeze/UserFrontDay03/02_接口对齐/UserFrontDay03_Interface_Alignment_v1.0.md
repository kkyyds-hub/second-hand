## 1. 对齐目标

围绕 `市场浏览、详情、评论与举报`，冻结 Day03 需要承接的市场、收藏与评论相关接口入口，并明确当前还没有任何对应前端实现落地。

---

## 2. 接口清单

| 场景 | 接口 / 契约 | 计划口径 | 备注 |
|---|---|---|---|
| 市场商品列表 | `GET /user/market/products` | 作为列表页主入口，后续承接筛选、分页、空态和错误态。 | 查询参数细节待执行时按 DTO 冻结。 |
| 市场商品详情 | `GET /user/market/products/{productId}` | 作为详情页主入口。 | 不与“我的商品详情”混用。 |
| 商品评论列表 | `GET /user/market/products/{productId}/reviews` | 作为详情页评论区主入口。 | 只冻结评论列表，不默认等于评价提交已完成。 |
| 商品举报 | `POST /user/market/products/{productId}/report` | 作为举报提交入口。 | 需记录失败提示、重复提交和留证方式。 |
| 收藏商品 | `POST /user/favorites/{productId}` | 作为详情页 / 列表页收藏动作入口。 | 不把收藏状态硬编码到市场详情响应里。 |
| 取消收藏 | `DELETE /user/favorites/{productId}` | 作为取消收藏动作入口。 | 需保留失败态与重复点击处理策略。 |
| 收藏列表 | `GET /user/favorites` | 作为收藏夹页主入口。 | 与市场列表共享鉴权基线，但页面归属独立。 |
| 收藏状态 | `GET /user/favorites/{productId}/status` | 作为详情页 / 卡片收藏按钮初始态入口。 | 需要与列表 / 详情渲染一致。 |
| 创建评价 | `POST /user/reviews` | 作为“评论提交”预留入口。 | 若执行阶段受订单前置限制，应明确写 `阻塞` 或拆到后续 Day。 |
| 我的评价 | `GET /user/reviews/mine` | 作为“我的评价入口”预留入口。 | 当前先冻结入口，不虚报页面已存在。 |

---

## 3. Day03 接口对齐原则

1. `market`、`favorite`、`review` 至少拆成独立 API 模块，不允许页面直接手写接口 URL。
2. 列表、详情、收藏状态、举报、评论都必须复用 Day01 的鉴权与 401 规则。
3. 收藏状态以显式接口为准；在接口未确认前，不把详情返回值假设为收藏真值来源。
4. 举报、评论等显式用户动作必须预留失败提示和留证说明，不能只记成功路径。
5. 若评价提交需要订单完成等前置条件，必须保留为依赖或阻塞，不能硬写为 Day03 已闭环。
