---

## 1. 起点结论

- 结论：`UserFrontDay05 应作为 买家订单、支付、取消、确认收货与售后发起 的正式入口。`
- 判定依据：
  1. Day03 的市场基线与 Day04 的商品状态，是买家订单主链继续推进的前置条件。
  2. 后端已存在 `OrdersController` 与 `AfterSaleController`，而用户端还没有买家订单页与售后入口。
  3. 若不单独建档，下单、支付、取消、确认收货与售后发起很容易和卖家履约链混写。

---

## 2. 做什么

1. 冻结买家订单列表 / 详情 / 下单 / 支付 / mock 支付回调 / 取消 / 确认收货 / 买家售后申请与纠纷发起的契约、API 模块、路由与回填口径。

---

## 3. 不做什么

1. 不提前吸收卖家发货、物流、卖家售后处理、钱包积分
2. 不把 mock 支付写成真实第三方支付已接通。

---

## 4. 范围拆解

| 范围分类 | 归属内容 | 处理方式 |
|---|---|---|
| 买家订单主链 | `OrdersController` 买家链路 | 作为 Day05 主链。 |
| 支付与 mock 回调 | `pay`、`pay/mock` | 明确演示支付与 mock 边界。 |
| 买家售后 / 纠纷发起 | `AfterSaleController` buyer 入口 | 卖家处理留给 Day06。 |

| 非 Day05 范围 | 卖家履约、钱包积分、系统通知 | 留给 Day06 以后处理。 |
