# UserFrontDay05 前后端接口对齐

- 日期：`2026-03-18`
- 文档版本：`v1.0`

---

## 1. 对齐目标

围绕 `买家订单、支付、取消、确认收货与售后发起` 冻结本日需要承接的接口入口，并明确当前还没有任何对应前端实现落地。

---

## 2. 接口清单

| 场景 | 接口 / 契约 | 计划口径 | 备注 |
|---|---|---|---|
| 买家订单列表 | `GET /user/orders/buy` | 作为买家订单列表页主入口。 | 查询字段待执行时确认。 |
| 订单详情 | `GET /user/orders/{orderId}` | 作为买家订单详情主入口。 | 若需要物流入口，只记入口，不抢 Day06 主链。 |
| 创建订单 | `POST /user/orders` | 作为下单提交入口。 | 字段级映射待执行时补充。 |
| 买家支付 | `POST /user/orders/{orderId}/pay` | 作为支付按钮主入口。 | 当前是演示支付口径。 |
| mock 支付回调 | `POST /user/orders/{orderId}/pay/mock` | 作为 Day05 联调辅助入口。 | 必须保留“非真实支付网关”说明。 |
| 取消订单 | `POST /user/orders/{orderId}/cancel` | 作为买家取消入口。 | 执行时补失败态说明。 |
| 确认收货 | `POST /user/orders/{orderId}/confirm-receipt` | 作为买家确认收货入口。 | 与 Day06 发货链共享状态机。 |
| 售后申请 / 纠纷 | `POST /user/after-sales`、`POST /user/after-sales/{afterSaleId}/dispute` | 作为 buyer 售后入口。 | 卖家处理不在 Day05。 |

---

## 3. 对齐原则

1. API 模块先分域规划，不把不同角色链路混写。
2. 继续复用 Day01 的鉴权、401 与 `authentication` 请求头规则。
3. 所有失败态、空态、阻塞说明都要保留在文档与后续回填里。
4. 字段级不确定项一律写成“待执行时确认”。
