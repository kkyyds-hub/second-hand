# UserFrontDay04 范围冻结（Scope Freeze）

- 日期：`2026-03-18`
- 文档版本：`v1.0`
- 当前状态：`计划中（v1.0 已建档，执行未开始）`
- 当日主题：`用户商品管理与卖家工作台扩展`

---

## 1. 起点结论

- 结论：`UserFrontDay04` 应作为第一批计划中的卖家域正式入口。
- 判定依据：
  1. Day01 只拥有首页卖家摘要基线，还没有卖家工作台和商品管理前端面；
  2. 后端已存在 `SellerController` 与 `UserProductController`，说明卖家域具备真实接口面；
  3. 若不把卖家域独立建档，商品管理和状态流转很容易被误塞到 Day01 或 Day05 之后的订单域里。

---

## 2. Day04 要做什么

1. 冻结卖家工作台摘要卡片、导航与“我的商品”入口。
2. 冻结用户商品列表、详情、发布、编辑、删除的页面 / API / 回填边界。
3. 冻结商品状态流转：下架、重新提交、兼容 `on-shelf`、撤回。
4. 规划 Day04 的 API 模块、路由扩展、联调准备与回填台账。
5. 明确 Day04 只负责卖家工作台与商品管理，不越级吸收订单、钱包、积分等后续域。

---

## 3. Day04 不做什么

1. 不把 Day01 首页 seller summary 直接写成“卖家工作台已完成”。
2. 不把买家订单、卖家订单、物流、钱包、积分提前写进 Day04。
3. 不在语义未确认前，把 `on-shelf` 写成“直接上架到 on_sale”。
4. 不把 Day04 的问题写回 `demo-admin-ui`。

---

## 4. Day04 范围拆解

| 范围分类 | 归属内容 | Day04 处理方式 |
|---|---|---|
| 卖家工作台扩展 | `GET /user/seller/summary` + 工作台入口 | 作为 Day04 主链，负责从首页摘要升级到独立卖家工作台入口。 |
| 用户商品管理 | `GET/POST/PUT/DELETE /user/products...` | 作为 Day04 主链，拆清列表、详情、表单与删除流程。 |
| 商品状态流转 | `off-shelf / resubmit / on-shelf / withdraw` | 作为 Day04 主链，明确按钮文案、状态语义与风险说明。 |
| 沿用基线 | Day01 的 auth/session/router 基线、现有 `seller.ts` 摘要接口 | 只做接棒说明，不重写 Day01 结论。 |
| 非 Day04 范围 | 订单、支付、物流、钱包、积分 | 留给 Day05 以后处理。 |

---

## 5. Day04 未来验收口径

> 以下是 Day04 真正进入执行后才需要满足的收口口径；当前 v1.0 仅完成计划建档，不代表已经满足。

1. 卖家工作台、用户商品管理、商品状态流转三组子流都已有明确 owner、接口口径和回填入口；
2. 卖家工作台不再只是 Day01 首页摘要，而是具备独立导航和商品管理入口；
3. 商品列表 / 详情 / 发布 / 编辑 / 删除 / 状态按钮的边界清晰可落地；
4. `on-shelf` 的兼容语义与 `resubmit` 关系已被明确记录，不再误导成直接上架；
5. 构建、运行、联调状态都能在 `05_进度回填` 中找到真实证据。
