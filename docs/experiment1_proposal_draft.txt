《二手商品交易与运营管理平台项目建议书》

一、项目基本信息
项目名称：二手商品交易与运营管理平台（Second-Hand）
项目类型：Web 应用系统（前后端分离）
项目提出人：毛垲（2303010213）
项目承担方式：课程实验项目，以个人开发为主，按标准软件工程流程推进
项目组成：用户端 demo-user-ui、管理端 demo-admin-ui、后端服务 demo-service
项目周期：12 周
预计开发规模：1—3 人

二、项目背景与问题描述
随着网络交易方式的普及，二手商品流通已经成为日常生活中的常见需求。无论是书籍、数码产品、生活用品，还是具有短期使用属性的物品，都存在“使用周期短、闲置率高、再利用价值高”的特点。如果没有一个规范的平台，用户往往只能通过微信群、QQ群、朋友圈或熟人介绍等方式完成交易。这种方式虽然门槛低，但在信息管理、交易安全、售后处理和平台治理方面存在明显不足。

第一，信息发布渠道分散，商品信息无法统一沉淀。卖家把商品发在不同群聊之后，买家很难快速搜索到需要的物品，也无法按照分类、价格、状态等条件进行筛选，导致信息匹配效率低。

第二，交易信任不足。传统的私下交易往往缺少账号体系、信用记录和举报机制，买卖双方无法准确判断对方是否可靠，一旦出现虚假描述、恶意拖延或交易纠纷，缺少可追溯的证据和处理流程。

第三，交易流程不完整。很多非平台化交易只有“看图—联系—转账”几个步骤，缺少下单、支付、发货、确认收货、评价、售后等闭环能力，容易在付款、物流、退款等环节出现争议。

第四，缺少后台管理与风控支撑。对于一个真实可运营的二手平台来说，仅有用户前台是不够的，还需要有商品审核、违规处理、信用调整、售后仲裁、运营统计、消息补发和任务运维等后台能力，以保证平台秩序和服务质量。

结合当前项目的实际情况，本仓库已经形成了较清晰的系统雏形：后端包含用户侧与管理侧控制器，覆盖认证、商品、订单、支付、收藏、评价、售后、信用、钱包、消息、违规治理等业务；前端拆分为用户端和管理端两个工程，管理端已经具备工作台、用户与商家管理、商品审核、纠纷与违规、运维中心等页面基础。因此，本实验拟以当前项目为核心，整理并完善为一个“二手商品交易与运营管理平台”项目建议书，用软件工程方法说明其建设必要性与实施路径。

三、项目建设目标
1. 建立统一的二手商品信息发布与浏览平台，实现商品展示、筛选、详情查看等基础交易入口。
2. 实现完善的用户认证体系，支持手机号注册、邮箱注册、邮箱激活、账号登录以及个人资料维护。
3. 面向卖家提供商品发布、编辑、上架、下架、重提审等能力，提升卖家经营效率。
4. 面向买家提供订单查询、支付、物流跟踪、确认收货、收藏、评价等完整交易支持。
5. 建立售后与纠纷处理机制，支持举报、仲裁、信用记录、违规治理等功能，提高平台交易安全性。
6. 建设管理后台，实现商品审核、用户封禁与解封、信用调整、统计看板、运维任务执行、消息补发等运营管理能力。
7. 通过前后端分离的系统架构，提高系统可维护性、扩展性和后期迭代效率，为后续功能扩展打下基础。

四、项目可行性分析
1. 技术可行性
当前项目已经具备较成熟的技术基础。前端采用 Vue 3、TypeScript、Vite 构建用户端与管理端，能够较好支持组件化开发、页面路由管理和接口联调；后端采用 Spring Boot 2.7.3 与 Java 17，结合 MyBatis-Plus、PageHelper、Druid、JWT、Knife4j 等技术实现业务接口、权限认证和文档支持；数据层可使用 MySQL 存储核心交易数据，Redis 支持缓存，MongoDB 可用于站内消息等场景，RabbitMQ 可用于异步消息与任务解耦。从当前仓库结构看，项目已经具备继续开发和整合的现实条件，因此技术路线明确、实现风险可控。

2. 经济可行性
本项目主要采用开源技术栈，开发工具和运行环境获取成本较低，适合课程设计和中小型平台实验。开发阶段可在普通个人电脑和本地开发环境中完成，部署阶段使用常规云服务器即可满足基本需求。由于系统以学习实践和平台验证为目标，不需要过高的商业化投入，因此整体经济成本较低，具备可实施性。

3. 运行可行性
系统采用浏览器访问模式，用户端和管理端界面清晰，使用门槛较低。普通用户可通过登录、浏览、下单、支付、查看订单等流程完成交易活动；平台管理员可通过后台进行审核、风控和运维处理。前后端分离也便于后续定位问题、更新模块和持续优化，因此运行维护较为方便。

4. 合规与安全可行性
平台涉及用户账号、订单、支付、地址、联系方式等信息，因此必须重视安全与合规。当前项目已引入 JWT 鉴权、角色区分、审核与举报机制，具备进一步加强权限控制、日志审计、敏感信息保护的基础。项目本身不涉及高污染、高能耗硬件环境，对生态环境影响较小；但在法律与平台治理层面，需要明确用户协议、违规处罚、隐私保护和交易责任边界，这些都可以通过后台审核、信用记录和操作留痕进行支撑。

五、项目范围与主要功能
1. 用户认证与账户中心模块
包括手机号注册、邮箱注册、邮箱激活、密码登录、个人资料维护、头像上传、绑定手机/邮箱、密码修改等功能。

2. 商品与卖家中心模块
包括商品发布、编辑、删除、上下架、重新提交审核、卖家经营摘要、商品详情查看等功能，使卖家能够完成商品管理。

3. 市场浏览与交易订单模块
包括市场商品浏览、商品详情、订单创建、支付、模拟支付、卖家发货、物流跟踪、确认收货、买家订单与卖家订单查询等功能，形成交易主链路。

4. 互动与增值服务模块
包括商品收藏、订单消息、系统通知、用户评价、积分总额与积分流水、钱包余额与提现申请等功能，用于提升用户体验和平台活跃度。

5. 售后与风控治理模块
包括售后申请、卖家处理、升级纠纷、商品举报、信用分记录、用户违规记录等功能，用于保障交易秩序。

6. 平台运营管理模块
包括管理端首页经营概览、商品待审列表、商品通过/驳回、强制下架、用户封禁/解封、信用调整、纠纷与违规处理、统计分析、运维任务执行、Outbox 消息补发等功能。

7. 非功能性范围
系统需要满足基本的安全性、可维护性、可扩展性和可测试性要求；同时在本阶段暂不重点追求复杂推荐算法、超大规模并发架构和商业级真实支付对接，而是以核心业务闭环与课程实验落地为主。

六、项目主要利益相关者
1. 项目提出人：毛垲，负责项目选题、方案组织与实验文档整理。
2. 开发者：负责前端、后端、数据库与接口联调实现，是项目落地的核心执行者。
3. 普通用户（买家/卖家）：平台最终使用者，直接决定系统是否真正有使用价值。
4. 平台管理员：负责商品审核、用户管理、违规处置、售后仲裁、运营监控等工作。
5. 指导教师：关注项目是否符合课程设计要求，是否体现软件工程分析、设计与实施过程。
6. 测试与维护人员：负责验证功能正确性、记录缺陷并保障系统后续可运行。

七、项目组织结构
虽然课程实验阶段以个人开发为主，但仍应按照标准项目组织方式划分角色，便于后续扩展与管理：

项目负责人
├── 需求分析与文档设计
├── 前端开发（用户端 / 管理端）
├── 后端开发与数据库设计
└── 测试与验收

这种组织方式的意义在于：即使由一人兼任多个角色，也能按照规范区分需求、设计、编码、测试和文档工作，避免开发过程混乱。

九、项目风险分析
1. 技术集成风险
本项目功能较多，涉及前端、后端、数据库、缓存、消息队列等多种技术。如果接口约定不统一，容易出现联调困难、字段不匹配或异步流程不一致的问题。应通过接口文档、统一返回格式和分阶段联调降低风险。

2. 进度风险
课程实验时间有限，而项目模块较多，若前期计划不清晰，容易出现后期功能堆积、文档来不及整理等问题。应将“核心链路优先”作为原则，先完成登录、商品、订单、审核等主功能，再逐步补充增强功能。

3. 数据与安全风险
系统涉及账号、地址、联系方式、订单和支付信息，一旦权限控制不足或数据校验不严，可能出现信息泄露、越权访问、恶意刷接口等问题。应加强鉴权、输入校验、日志记录和异常处理，必要时增加敏感操作审计。

4. 外部依赖风险
项目中部分功能依赖邮件激活、短信验证码、缓存、中间件和异步任务。如果开发环境中 Redis、MongoDB、RabbitMQ 或邮件服务配置不完整，可能影响部分功能演示。对此可采用 mock 机制和降级方案，保证核心流程可演示。

5. 需求变化风险
随着实验推进，可能新增界面细节优化、统计展示或业务规则调整。如果频繁变更需求，会影响已完成模块的稳定性。应在需求分析阶段尽量明确业务边界，对变更进行记录和评估。

3. 数据库设计说明与核心实体结构说明文档。
4. 主要接口说明文档与联调记录。
5. 系统测试报告、缺陷修复记录与功能验收说明。

十一、项目成功标准
1. 系统能够完成“注册/登录—商品发布—浏览下单—支付发货—确认收货—评价/售后—后台审核治理”的核心业务闭环。
2. 用户端、管理端与后端接口能够正常联通，主要页面与接口无阻塞性错误。
3. 系统具备基本的权限控制、异常处理和日志留痕能力，关键数据能够正确保存。
4. 项目文档完整，能够清楚说明需求背景、设计思路、功能模块、实施计划与测试结果。
5. 系统界面与交互能够满足课程实验演示要求，具有一定的实用价值和扩展空间。

十二、结论
综上所述，二手商品交易与运营管理平台具有明确的现实需求和较高的实践价值。它不仅能够解决二手交易中信息分散、信任不足、售后缺失、平台治理薄弱等问题，还能够完整体现软件工程中项目立项、需求分析、系统设计、编码实现、测试验收和文档整理的全过程。结合当前仓库的技术基础与功能结构，本项目已经具备较好的落地条件。通过本次实验，不仅可以形成一份结构完整的项目建议书，也能为后续系统开发、功能完善和课程答辩提供清晰依据，因此该项目具有较高的可行性和实施意义。
