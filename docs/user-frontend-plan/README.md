# 用户端前端规划主入口

> 创建日期：`2026-03-17`
> 适用范围：仓库级用户端前端规划
> 当前状态：`规划已建立；用户端独立主工程仍是 blocker`
> 说明：本目录是**规划入口**，不是未来用户端 freeze 主入口

---

## 1. 这套规划解决什么问题

1. 把“用户端前端到底先做什么”从零散想法收敛成统一入口。
2. 把“按后端模式推进前端”具体化为可执行的阶段、证据层和收口方式。
3. 在 `demo-user-ui` 尚未存在时，先把前置条件、边界、推进顺序和未来四技能工作流定义清楚。
4. 避免把管理端冻结体系、用户端规划、实际页面开发和后续联调混在一起。

---

## 2. 当前仓库结论

### 2.1 已确认的现实

1. 当前仓库里唯一明确的前端主工程是 `demo-admin-ui`。
2. 截至 `2026-03-16`，管理端 `demo-admin-ui/docs/frontend-freeze/FrontDay10/README.md` 已完成“演示版冻结与移交”收口。
3. `demo-service/src/main/java/com/demo/controller/user/` 下已经存在较完整的用户端后端接口面，但仓库内还没有对应的用户端前端主工程。
4. 因此，本轮能落地的是**规划体系**，不是用户端四技能本体，也不是用户端 freeze 骨架本体。

### 2.2 本轮明确不做

1. 不在不存在真实工程的前提下硬建 `write-vue-user-ui` 等空壳技能。
2. 不在根目录 `docs/` 下伪造用户端 freeze 体系。
3. 不拿管理端 `FrontDay10` 的冻结状态承接用户端工作。
4. 不把用户端推进误写成“先挑几个页面做掉”。

---

## 5. 这版规划的核心判断

这版前端计划的核心不是“先写哪些页面”，而是：

1. 先确定用户端是一个独立主工程；
2. 再建立用户端自己的四技能入口；
3. 再用和后端一致的阶段化收口方法推进；
4. 最后才是具体页面开发顺序。

这样做的好处是：

- 不会把 user/admin 边界做混；
- 不会把页面开发、联调、freeze 混成一锅；
- 后续每一轮都能判断“该用哪个技能、该回填到哪一层、该算 code-confirmed 还是 runtime-confirmed”。
