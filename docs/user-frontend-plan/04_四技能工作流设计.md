# 04_四技能工作流设计

> 规划日期：`2026-03-17`
> 说明：本文件定义未来用户端四技能的职责边界；本轮不创建技能实体

---

## 1. 为什么要做四技能，而不是只做一个大而全入口

用户端前端后续会同时涉及：

- 页面实现
- 接口对齐
- 路由与鉴权
- 真实联调
- freeze 文档回填

如果所有任务都用一个大而全入口，短期方便，长期会出现：

1. 纯页面任务被迫加载过多上下文
2. 纯验证任务和纯文档任务没有清晰边界
3. skill 触发条件越来越模糊
4. 一次任务到底该改代码、跑验证还是回填文档很难判断

因此，用户端未来也应采用“四技能工作流”。

---

## 2. 未来建议的四技能命名

在 `demo-user-ui` 主工程落地后，再创建以下技能：

1. `write-vue-user-ui`
2. `verify-demo-user-frontend-runtime`
3. `manage-user-frontend-freeze-docs`
4. `drive-demo-user-ui-delivery`

---

## 3. 四技能职责边界

| 技能名 | 主要职责 | 适用场景 | 不适用场景 |
|---|---|---|---|
| `write-vue-user-ui` | 只负责用户端前端代码实现与重构 | 页面、组件、路由、样式、已知契约下的 API 包装 | 真实联调闭环、freeze 治理、跨端交付 |
| `verify-demo-user-frontend-runtime` | 只负责用户端运行验证与证据留存 | smoke、回归、关键链路验证、pass/fail/blocker 分类 | 主要任务是写页面或改文档 |
| `manage-user-frontend-freeze-docs` | 只负责用户端 freeze 文档治理与回填 | 范围冻结、进度回填、覆盖矩阵维护、handoff 文档 | 真实代码实现、运行验证、跨端联调定位 |
| `drive-demo-user-ui-delivery` | 负责跨边界推进 | 同时涉及代码、契约、验证、freeze 回填的任务 | 纯页面、纯验证、纯文档任务 |

---

## 4. 第一阶段为什么优先用 `drive-demo-user-ui-delivery`

在 `demo-user-ui` 刚建好、边界还未稳定时，一个用户端任务往往会同时跨越：

1. 路由和布局壳
2. request 层和 token 头
3. 页面与 API 绑定
4. 构建与运行验证
5. freeze 状态升级

因此，第一阶段的默认推荐入口应是：

- `drive-demo-user-ui-delivery`

待边界稳定后，再把任务拆回三个更窄的技能。

---

## 5. 技能创建的前提条件

只有满足以下条件，才应该真正创建这四个技能：

1. 仓库内已存在明确的 `demo-user-ui`
2. 该工程已具备包管理与构建入口
3. 该工程已具备 `router / pages / api / request` 入口
4. 用户端和管理端的边界能用真实路径表达
5. 用户端 freeze 根入口已能挂到 `demo-user-ui/docs/frontend-freeze/`

在这些条件满足之前，本文件只定义“未来怎么建”，不定义“现在就建”。

---

## 6. 推荐的任务分流规则

### 6.1 默认分流顺序

1. 如果任务同时涉及代码、接口、验证、文档：`drive-demo-user-ui-delivery`
2. 如果任务只是页面或组件实现：`write-vue-user-ui`
3. 如果任务只是运行态验证：`verify-demo-user-frontend-runtime`
4. 如果任务只是 freeze 治理或回填：`manage-user-frontend-freeze-docs`

### 6.2 一条稳定规则

如果执行中发现任务已经跨出原技能边界，应升级到：

- `drive-demo-user-ui-delivery`

而不是继续用一个过窄技能硬扛到结束。
