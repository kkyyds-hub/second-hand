## 2. 为什么现在不能直接建立用户端四技能和 freeze

用户端四技能和 freeze 体系都必须绑定到真实工程面，否则会出现以下问题：

1. 技能无法指向真实的 `src/pages`、`src/api`、`src/utils/request.ts`。
2. 运行验证技能无法指向真实 dev server、构建脚本和路由入口。
3. freeze 文档无法绑定真实页面、真实 API 模块、真实 runtime 证据。
4. 很容易退化成“只围着后端接口写前端计划”，最后 user/admin 边界混乱。

所以，本轮正确动作是：

- 先把工程前置条件和边界写清楚；
- 等用户端主工程落地后，再创建技能和 freeze。

## 3. 用户端主工程的最小定义

建议未来用户端采用独立工程：

- 建议路径：`demo-user-ui`
- 建议形态：**buyer / seller 单工程**

### 3.1 采用 buyer / seller 单工程的原因

1. 当前后端接口统一挂在 `/user/**` 下，天然更接近单工程单会话。
2. 买家、卖家、个人中心、消息、订单之间共享登录态和用户身份。
3. 单工程更容易统一 `request` 层、路由守卫、布局壳和 freeze 证据口径。
4. 在项目现阶段，先拆成两个前端工程会过早引入边界和协同成本。

### 3.2 最小入口文件要求

未来至少需要具备以下文件，才算“用户端主工程成立”：

- `demo-user-ui/package.json`
- `demo-user-ui/src/main.ts`
- `demo-user-ui/src/router/index.ts`
- `demo-user-ui/src/layouts/UserLayout.vue`
- `demo-user-ui/src/pages/...`
- `demo-user-ui/src/api/...`
- `demo-user-ui/src/utils/request.ts`

### 3.3 推荐补充入口

- `demo-user-ui/src/stores/...`
- `demo-user-ui/src/style.css`
- `demo-user-ui/.env.*`
- `demo-user-ui/docs/frontend-freeze/README.md`（仅在工程落地后创建）

## 4. user / admin 边界规则

### 4.1 工程边界

- 管理端工程固定为：`demo-admin-ui`
- 用户端工程规划为：`demo-user-ui`

两端前端代码、验证、freeze、技能入口都必须分开维护。

### 4.2 文档边界

- 管理端冻结入口固定为：`demo-admin-ui/docs/frontend-freeze/README.md`
- 用户端未来冻结入口固定为：`demo-user-ui/docs/frontend-freeze/README.md`
- 本目录 `docs/user-frontend-plan/` 只作为**立项与规划入口**，不作为未来 freeze 主入口

### 4.3 接口边界

用户端默认只消费：

- `demo-service/src/main/java/com/demo/controller/user/` 下的 `/user/**` 能力

如果后续需要接入非 `/user/**` 的共享接口，例如支付、上传配置、公共资源或第三方回调辅助接口，必须在用户端的接口对齐文档中单独冻结，不允许默认混入。

### 4.4 鉴权边界

根据 `demo-service/src/main/resources/application.yml`，当前后端已区分：

- 管理端 token header：`token`
- 用户端 token header：`authentication`

因此未来用户端前端必须建立独立 `request` 层，不能直接复用管理端 `demo-admin-ui` 的鉴权口径。

---

## 5. 前端按后端模式推进的具体含义

用户端前端采用和后端一致的 4 个治理原则：

1. **先冻结范围，再做实现**
2. **每个阶段都必须有证据层**
3. **按业务域推进，不按零散页面推进**
4. **每个阶段都要能收口和移交**

这 4 条是用户端规划的稳定规则，未来即使具体 Day 编号调整，也不应改变。
