# 03_证据分层与冻结准备

> 规划日期：`2026-03-17`
> 说明：本文件定义的是未来用户端 freeze 的规则，不代表 freeze 已经建立

---

## 1. 前端证据层必须和后端一样明确

后端已经有“执行复现步骤 / 执行记录 / 冻结与移交”的收口思路。  
用户端前端未来也必须有自己的证据层，只是证据类型不同。

### 1.1 用户端前端至少需要三类证据

1. **构建证据**
   - `npm run build` 结果
   - 关键 TypeScript / Vue 编译结果

2. **运行验证证据**
   - 浏览器访问结果
   - 关键请求响应
   - 关键交互截图或结构化运行记录

3. **联调回填证据**
   - 页面与接口映射
   - 实际通过项 / 失败项 / blocker
   - 哪些仍是 mock / fallback / 兼容映射

---

## 2. 建议采用的证据等级

未来用户端前端统一建议按以下等级描述状态：

| 等级 | 含义 | 允许表达 | 不允许表达 |
|---|---|---|---|
| `planned` | 仅有范围和计划 | 已规划 | 已完成 |
| `code-confirmed` | 代码已落地并人工核对 | 代码已确认 | 联调完成 |
| `build-confirmed` | 构建已通过 | 构建已确认 | 运行通过 |
| `runtime-confirmed` | 真实运行已观察到 | 运行态已确认 | freeze 完成 |
| `freeze-closed` | 范围、证据、遗留项、移交均已收口 | 已冻结并回填 | 无依据升级表述 |

## 3. 用户端未来 freeze 结构

当且仅当 `demo-user-ui` 主工程已经存在时，再建立：

- `demo-user-ui/docs/frontend-freeze/README.md`
- `demo-user-ui/docs/frontend-freeze/00_Business_Coverage_Matrix.md`
- `demo-user-ui/docs/frontend-freeze/UserFrontDay01/`
- `demo-user-ui/docs/frontend-freeze/UserFrontDay02/`
- ...
- `demo-user-ui/docs/frontend-freeze/UserFrontDay10/`

### 3.1 为什么不用 `FrontDayNN`

管理端已经使用：

- `demo-admin-ui/docs/frontend-freeze/FrontDay01~10`

为了明确区分 user/admin，用户端未来建议改用：

- `UserFrontDay01~10`

这样可以避免：

1. 管理端和用户端在口头沟通时混淆“Day09 / Day10”
2. 技能回填路径和覆盖矩阵路径混写
3. 迁移或 handoff 时无法一眼识别当前文档属于哪一端

### 3.2 每个 `UserFrontDayNN` 的建议结构

每个目录建议保持和管理端同构，但根路径换成 `demo-user-ui`，Day 名字换成 `UserFrontDayNN`：

1. `README.md`
2. `01_冻结文档/UserFrontDayNN_Scope_Freeze_v1.0.md`
3. `02_接口对齐/UserFrontDayNN_Interface_Alignment_v1.0.md`
4. `03_API模块/UserFrontDayNN_API_Module_Plan_v1.0.md`
5. `04_联调准备与验收/UserFrontDayNN_Joint_Debug_Ready_v1.0.md`
6. `05_进度回填/UserFrontDayNN_Progress_Backfill_v1.0.md`

## 4. freeze 回填顺序

未来用户端 freeze 更新时，建议保持和管理端一致的回填顺序：

1. 先看代码和运行证据
2. 先更新 `05_进度回填`
3. 再更新 `00_Business_Coverage_Matrix.md`
4. 再更新 `02_接口对齐` / `03_API模块` / `04_联调准备与验收`
5. 最后再更新 Day `README.md` 和根 `README.md`

---

## 5. 本轮和 freeze 的关系

本轮建立的是：

- 规划入口
- 阶段路线图
- 未来技能与 freeze 的设计规则

本轮**没有建立**：

- `demo-user-ui`
- 用户端四技能
- 用户端 `docs/frontend-freeze/`

这不是缺项，而是为了避免在真实工程缺失时制造空壳体系。
