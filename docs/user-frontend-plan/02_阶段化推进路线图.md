# 02_阶段化推进路线图

> 规划日期：`2026-03-17`
> 目标：把“先做哪些前端工作”收敛为阶段化业务域闭环

---

## 1. 总体推进方式

用户端前端不按“先挑几个页面做掉”推进，而按以下顺序推进：

1. **P0 前置阶段**：先把独立主工程建立起来
2. **UserFrontDay01~10**：在真实工程上按业务域推进
3. **每个 Day 都带范围、证据、回填、移交**

---

## 2. P0：独立主工程前置阶段

### 2.1 目标

在仓库中创建真实可运行的用户端主工程 `demo-user-ui`，使后续技能、代码、验证和 freeze 都能挂到真实工程上。

### 2.2 最小产物

1. `demo-user-ui/package.json`
2. `demo-user-ui/src/main.ts`
3. `demo-user-ui/src/router/index.ts`
4. `demo-user-ui/src/layouts/UserLayout.vue`
5. `demo-user-ui/src/pages/`
6. `demo-user-ui/src/api/`
7. `demo-user-ui/src/utils/request.ts`

### 2.3 退出标准

- 能安装依赖
- 能启动 dev server
- 能跑一次空壳 build
- 有最小可达路由壳
- 能向后端带上 `authentication` header

### 2.4 明确不做

- 不在 P0 阶段追求业务页面闭环
- 不在 P0 阶段建立完整 freeze
- 不在 P0 阶段大规模联调所有 `/user/**` 接口

## 4. 为什么第一步是“工程基建与鉴权壳”

不从收藏、订单或售后开始，而从工程基建与鉴权壳开始，原因是：

1. 订单、消息、售后都依赖稳定登录态与 request 口径。
2. 用户端的 `authentication` header 与管理端的 `token` header 已明确分离。
3. 没有独立布局壳、路由守卫、错误空态骨架，后续页面会反复返工。
4. 先把壳搭稳，后面每个业务域才能按 freeze 方式收口。

## 5. 业务域推进规则

### 5.1 每个业务域至少包含四层

1. 页面层：`src/pages/...`
2. API 层：`src/api/...`
3. request / state 层：`src/utils/request.ts`、`stores/...`
4. 文档与证据层：freeze / runtime / progress 回填

### 5.2 每个业务域都要同时回答

1. 这次做什么？
2. 这次不做什么？
3. 主要依赖哪些后端接口？
4. 证据怎么留？
5. 什么时候算收口？

### 5.3 状态口径

建议未来统一使用：

- `planned`
- `code-confirmed`
- `build-confirmed`
- `runtime-confirmed`
- `freeze-closed`

不允许把“代码写了”直接表述成“联调完成”或“冻结完成”。

---

## 6. 一条重要执行建议

在 `demo-user-ui` 刚建立、业务边界和页面分层还不稳定的阶段，实际推进建议优先用：

- `drive-demo-user-ui-delivery`

原因是第一阶段往往会同时碰到：

- 页面实现
- 接口对齐
- request 层调整
- 路由和鉴权
- 构建验证
- freeze 口径回填

这时直接用 `write-vue-user-ui` 会过窄，容易让任务在多个入口之间来回切换。
