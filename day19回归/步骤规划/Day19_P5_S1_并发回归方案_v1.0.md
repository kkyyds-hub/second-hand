## 1. 目标与范围

本步骤目标：验证高并发下核心业务语义是否稳定，重点不是“谁先成功”，而是“有没有重复副作用”。

本次范围：

1. 同订单重复支付
2. 同订单重复成功回调
3. 同用户重复封禁
4. 同用户重复解封
5. 发货超时任务并发抢同一条任务

---

## 2. 统一口径

1. 主成功：只有一个请求真正完成状态流转
2. 幂等成功：其余同向并发请求收敛为固定业务文案
3. 失败：出现非预期业务错误、HTTP 异常或最终状态不一致
4. 通过标准：无重复关单、无重复退款、无重复封禁、无状态反转

---

## 3. 执行参数冻结

1. 默认并发：`50`
2. 目标样本：支付 / 回调 / 封禁 / 解封 / 任务链路各 `1` 组
3. 验证输出：`P95`、冲突分流文案、最终 DB 状态

---

## 4. 预期结论

1. CAS 分流语义稳定，不会把同向并发误判成真实失败。
2. MQ / 任务 / DB 组合链路在并发下不产生重复副作用。
3. 最终产物可直接用于说明“我怎么验证幂等和最终一致性”。

---

## 5. DoD 映射

| DoD | 对应设计动作 |

| 冲突分流文案与日志口径稳定 | 统计 messageCounts 与固定业务文案 |
