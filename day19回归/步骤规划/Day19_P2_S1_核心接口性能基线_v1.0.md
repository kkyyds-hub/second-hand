## 1. 目标与口径

本步骤目标：为 Day19 建立可复跑、可对比、可解释的本地性能基线，覆盖三条核心链路：

1. 登录：`POST /user/auth/login/password`
2. 商品列表：`GET /user/market/products?page=1&pageSize=20`
3. 支付回调：`POST /payment/callback`

统一成功口径：

1. HTTP 层：`2xx`
2. 业务层：`Result.code = 1`
3. 错误率：不满足上述任一条件即计入错误

统一指标口径：

1. `P50/P95`：按请求级延迟分位数统计
2. `错误率`：`failedRequests / totalRequests`
3. `吞吐`：`totalRequests / durationSec`

## 6. 结果解释

1. 登录链路尾延迟显著高于其它两条链路：
   - `P95=449.26ms`，明显高于商品列表与回调。
   - 原因可解释：登录涉及用户查询、密码校验、JWT 签发与风控计数等写读混合逻辑。

2. 商品列表链路低延迟且稳定：
   - `P95=13.31ms`，分轮波动小。
   - 本次返回样本中列表数据 `total=0`，反映的是“空列表路径”成本，后续若用于预算需补“有数据页”样本。

3. 支付回调链路在本次样本下稳定：
   - `P95=20.00ms`，错误率 `0%`。
   - 本次为“幂等回调路径”基线，适合重复压测；若评估最重路径，需补“pending->paid 首次回调”专项样本。
