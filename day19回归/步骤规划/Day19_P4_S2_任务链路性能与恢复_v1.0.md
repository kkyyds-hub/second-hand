## 1. 目标与范围

本步骤目标：验证 `ship-timeout / refund / ship-reminder` 三类任务在压力下的处理稳定性与恢复闭环。

本次范围：

1. 多轮执行 `run-once`，统计每轮 `success / elapsedMs / 吞吐(task/s)`。
2. 覆盖 `trigger-now / reset` 手工补偿路径。
3. 覆盖至少一条自动恢复样本。
4. 产出容量建议与重试间隔建议。

---

## 2. 统一口径

1. 吞吐：`successCount / elapsedSec`
2. 处理成功：任务推进到目标终态，且与接口返回结果交叉校验一致
3. 手工恢复耗时：从补偿动作触发到下一次 `run-once` 完成进入终态
4. 自动恢复耗时：从首次回收触发到等待重试窗口后再次成功

---

## 3. 执行参数冻结

1. `BatchSizes = 10 / 25 / 50 / 100`
2. 每个 `limit` 轮次：`3`
3. 手工补偿样本：每类任务 `1` 条
4. 自动恢复样本：`ship-reminder stale RUNNING` `1` 条
5. 补充吞吐验证：`limit=150 / 200`

---

## 4. 预期结论

1. 三类任务都能给出可解释的吞吐曲线。
2. `trigger-now / reset` 不只是接口可用，而是恢复路径可量化。
3. 最终能沉淀“limit 推荐值”和“重试间隔建议”。
