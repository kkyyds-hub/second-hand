## 1. 这次 P5-S1 的目标是什么

本次 `P5-S1` 的目标不是单纯“多打几次并发请求”，而是验证以下三件事是否在高并发下仍然成立：

1. **业务语义稳定**：同一业务动作在并发冲突下仍能保持幂等，不产生重复副作用。
2. **分流口径稳定**：CAS 命中、CAS 未命中、幂等返回、冲突失败，这些分支的返回文案和日志口径不能漂移。
3. **最终一致性稳定**：即使过程里有异步消息、任务链路、回调竞争，最后数据库状态也要落在正确结果上。

本次覆盖的链路包括：

1. 支付链路
2. 支付成功回调链路
3. 用户封禁 / 解封链路
4. 发货超时任务处理链路

## 2. 这次执行过程中我实际遇到了哪些问题

### 2.1 问题一：支付回调并发下出现“假失败”

首轮并发回归时，`S2_DUPLICATE_CALLBACK` 的结果不是稳定的“1 次成功 + 49 次幂等”，而是出现了如下漂移：

- `支付回调处理成功 x1`
- `订单已支付，回调幂等成功! x42`
- `支付回调处理失败，订单状态不允许支付：pending x7`

这说明有 7 个请求并不是业务真的失败，而是**在并发竞争里误判成失败**了。

### 2.2 问题二：封禁 / 解封并发下出现错误的 CAS_CONFLICT

`S3_DUPLICATE_BAN` 和 `S4_DUPLICATE_UNBAN` 在首轮复跑时也不稳定，表现为：

- `成功 x1`
- `幂等 x42`
- `用户状态已变化，请刷新后重试 x7`

理论上这 7 次也不该失败。因为这是**同向并发**：

- 多个请求同时“封禁同一个用户”
- 多个请求同时“解封同一个用户”

其中只有 1 个请求需要真正完成状态流转，其余请求都应该收敛为幂等，而不是冲突失败。

### 2.3 问题三：发货超时任务与异步库存回写发生“状态反转”

`S5_SHIP_TIMEOUT_TASK_RACE` 暴露的是一个更实质性的业务问题：

1. 发货超时任务已经把订单正确关闭为 `cancelled`
2. 商品也已经因为订单取消被释放回 `on_sale`
3. 但随后旧的 `ORDER_CREATED` 库存更新消息又被消费
4. 消费者再次把商品写回了 `sold`

也就是说：

- **订单最终是真取消**
- **商品最终却是假锁死**

这是典型的“异步旧消息覆盖当前业务真相”的最终一致性问题。

### 2.4 问题四：执行环境存在“多实例共享 DB / MQ”的干扰风险

回归过程中，本地同时存在过：

1. `8080` 实例
2. `18080` 实例

它们共享同一套本地数据库和 RabbitMQ。  
这会带来一个现实风险：**两个实例可能同时消费消息、同时跑任务、同时影响同一批测试数据**。

这不是代码 bug，但它会影响回归结果的可解释性，所以中间我专门做了实例清理和重新指定基座服务的动作。

### 2.5 问题五：仓库里已有无关测试编译失败，阻断了全量 `mvn test`

在准备跑最小测试集时，发现仓库现有测试 `OutboxPublishJobFailureInjectionTest` 自身就有编译问题，和本次 P5-S1 修改无关，但会阻断完整 `mvn test`。

因此本次验证路径改成了：

1. `mvn compile`
2. 动态并发脚本复跑
3. 结果 JSON 校验

这不是功能风险，但属于本次回归中的一个“验证噪音点”。

## 3. 为什么会出现这些问题

### 3.1 支付回调为什么会误读到 `pending`

对应代码：`demo-service/src/main/java/com/demo/service/serviceimpl/OrderServiceImpl.java`

原因核心是：**事务隔离级别 + 并发写入时序**。

原逻辑是：

1. 先执行 `pending -> paid` 的 CAS 更新
2. 如果 `rows == 0`，就再次查询订单当前状态
3. 再根据最新状态决定是“幂等成功”还是“非法状态”

这个思路本身是对的，但实际运行在事务里时，如果使用数据库默认的 `REPEATABLE_READ`：

1. 当前线程一开始读到的是 `pending`
2. 另一个线程先一步成功把订单改成了 `paid`
3. 当前线程 CAS 更新失败
4. 当前线程再次查询时，**仍可能继续读到自己事务快照里的旧 `pending`**

于是本该进入“幂等成功”的分支，结果却被误导到“状态不允许支付：pending”。

### 3.2 封禁 / 解封为什么会出现错误冲突

对应代码：`demo-service/src/main/java/com/demo/service/serviceimpl/UserServiceImpl.java`

问题本质和支付回调非常类似，也是**事务内的二次查询没有稳定读到最新提交结果**。

具体表现为：

1. 第一个请求把用户从 `active -> banned` 或 `banned -> active` 成功更新
2. 其他请求 CAS 更新失败
3. 失败后再次查询用户状态时，仍可能看到旧状态
4. 代码认为“既不是目标状态，也不是我期待的最终结果”，于是分流为 `CAS_CONFLICT`

但这类场景其实不是冲突，而是**同向并发下的幂等命中**。

### 3.3 发货超时任务为什么会把商品状态搞反

对应代码：`demo-service/src/main/java/com/demo/mq/consumer/InventoryUpdateConsumer.java`

根因不是 SQL 幂等失效，而是**幂等 SQL 无法表达“历史消息不一定代表当前真相”**。

当时消费者只做了一件事：

1. 拿到 `productId`
2. 执行 `markProductSoldIfOnSale(productId)`

这个 SQL 确实是幂等的，但问题在于：

- 它没有确认“这个订单此刻是否仍然有效”
- 它只知道“曾经发生过订单创建”
- 它不知道“这笔订单后来已经取消了”

所以只要消息比关单链路晚到一步，它就会把已经释放回 `on_sale` 的商品重新打回 `sold`。

### 3.4 为什么执行环境会影响判断

因为本次场景里既有：

1. 同步接口请求
2. 异步 MQ 消费
3. 任务链路

如果两个应用实例同时连着同一个 DB / MQ：

- 一个实例可能发请求
- 另一个实例可能消费消息
- 两个实例都可能处理同一条任务

这样即使代码本身没问题，回归数据也可能出现“你以为是这个实例处理的，其实是另一个实例处理的”。

## 4. 这次我具体修改了哪些代码

### 4.1 修改文件一：`demo-service/src/main/java/com/demo/service/serviceimpl/OrderServiceImpl.java`

本次修改点：

1. 新增统一常量 `PAYMENT_CALLBACK_IDEMPOTENT_MESSAGE`
2. 把 `handlePaymentCallback(...)` 改为 `READ_COMMITTED`
3. 保留并强化 CAS 未命中后的“短轮询刷新”
4. 统一回调幂等返回文案

#### 为什么要这样改

1. **统一文案**  
   解决“同一语义返回两种字符串”的问题，保证回归统计、前端判断、日志检索一致。

2. **切换到 `READ_COMMITTED`**  
   解决在事务内 CAS 失败后再次查询仍读到旧快照的问题。

3. **保留短轮询**  
   即使使用 `READ_COMMITTED`，极端时序下首次重查也可能正好早于对方事务提交完成，短轮询可以把这个极短窗口收敛掉。

#### 解决了什么问题

把原来的：

- `成功 x1`
- `幂等 x42`
- `失败 x7`

稳定收敛成：

- `支付回调处理成功 x1`
- `订单已支付，回调幂等成功 x49`

#### 改完的好处是什么

1. 回调链路不再把真实成功误报成失败
2. 接口分流语义稳定
3. 第三方重复通知不会制造无意义告警
4. 更容易满足 P5-S1 的回归 DoD

---

### 4.2 修改文件二：`demo-service/src/main/java/com/demo/service/serviceimpl/UserServiceImpl.java`

本次修改点：

1. 给 `banUser(...)` 增加方法级 `READ_COMMITTED`
2. 给 `unbanUser(...)` 增加方法级 `READ_COMMITTED`
3. 把 CAS 失败后的短轮询窗口扩大到约 `600ms`

#### 为什么要这样改

1. `UserServiceImpl` 类上本身有事务，方法不覆盖时会沿用默认隔离级别
2. 并发下 CAS 失败后的再次查询可能仍看不到其他线程刚提交的结果
3. 过短的刷新窗口不足以覆盖实际回归里观测到的提交可见性抖动

#### 解决了什么问题

把原来的：

- `成功 x1`
- `幂等 x42`
- `CAS_CONFLICT x7`

稳定收敛成：

- `成功 x1`
- `幂等 x49`

分别体现在：

1. `S3_DUPLICATE_BAN`
2. `S4_DUPLICATE_UNBAN`

#### 改完的好处是什么

1. 同向并发不再被误判成冲突
2. 管理端操作语义更符合直觉
3. 减少误报“请刷新后重试”的噪音
4. 日志里真正的 `CAS_CONFLICT` 更有排查价值

---

### 4.3 修改文件三：`demo-service/src/main/java/com/demo/mq/consumer/InventoryUpdateConsumer.java`

本次修改点：

1. 在消费 `ORDER_CREATED` 时，如果 payload 带有 `orderId`，先回查当前订单状态
2. 如果订单不存在、状态异常、或已取消，则直接 `ACK + 标记 OK + 跳过库存回写`
3. 只有订单仍是有效态时，才继续执行 `markProductSoldIfOnSale(productId)`

#### 为什么要这样改

因为库存更新消息描述的是**历史事件**，而不是**当前真相**。

`ORDER_CREATED` 只能证明：

- 这个订单曾经创建过

但不能证明：

- 这个订单现在还有效
- 这个商品现在仍然应该保持 `sold`

如果订单已经被发货超时任务取消，那么商品的当前真相就是 `on_sale`。  
此时旧消息不能再覆盖最新状态。

#### 解决了什么问题

把 `S5_SHIP_TIMEOUT_TASK_RACE` 从：

- 订单 `cancelled`
- 商品却是 `sold`

修复为：

- 订单 `cancelled`
- 商品恢复 `on_sale`

#### 改完的好处是什么

1. 修复真正的最终一致性 bug
2. 避免历史消息污染当前状态
3. 让“订单取消 -> 商品释放”链路闭环完整成立
4. 避免卖家商品被错误锁死无法再次售卖

## 5. 为什么这些修改能把问题修掉

### 5.1 `READ_COMMITTED` 为什么有效

因为本次问题不是“CAS 设计错了”，而是“CAS 失败后的再次查询没有稳定读到最新提交结果”。

改成 `READ_COMMITTED` 之后：

1. 当前事务每次查询都基于已提交数据
2. 并发线程一旦提交成功
3. 当前线程后续查询就有机会读到最新状态

因此，原本误判的失败会回到正确的幂等分支。

### 5.2 短轮询为什么仍然有必要

单靠 `READ_COMMITTED` 还不一定完全覆盖极端时序：

1. 线程 A CAS 失败
2. 线程 A 马上重查
3. 线程 B 可能就在这一瞬间尚未提交完成

短轮询的意义就是：

- 不重复写
- 不扩大副作用
- 只给数据库一个很短的“状态变得可见”的时间窗口

这是一种很轻量的并发语义稳定化手段。

### 5.3 订单状态校验为什么能解决库存回写反转

因为消息是异步的，异步系统里必须承认一件事：

> 历史事件的到达顺序，不一定等于业务状态的最终顺序。

所以库存消费者不能只看“我收到了创建消息”，还必须看“这笔订单现在是不是仍然有效”。

引入这个判断之后：

1. 有效订单 -> 继续回写 `sold`
2. 已取消订单 -> 跳过回写

这样消费者行为就和当前业务真相重新对齐了。

## 8. 这次回归过程中有哪些经验教训

### 8.1 CAS 成功与否只是第一层，读隔离同样影响最终语义

很多并发 bug 不是因为 CAS 本身错了，而是因为：

- CAS 失败后的补充分流，读到了不该读的旧状态

所以“写的并发控制”与“读的可见性策略”必须一起设计。

### 8.2 幂等 SQL 不能替代业务真相判断

`markProductSoldIfOnSale` 这种 SQL 是幂等的，但它只解决：

- 重复执行不会无限改写

它解决不了：

- 这次执行本身是否还应该发生

异步消费者一定要校验“当前业务真相”，不能只信历史消息。

### 8.3 并发回归必须把“接口结果”和“最终落库结果”一起看

如果只看接口返回：

- 你可能以为系统已经通过了

如果只看数据库最终状态：

- 你可能漏掉分流文案漂移

这次 P5-S1 能最终闭环，是因为我同时看了：

1. 接口返回比例
2. 日志
3. 最终 DB 状态
4. 异步消息影响
