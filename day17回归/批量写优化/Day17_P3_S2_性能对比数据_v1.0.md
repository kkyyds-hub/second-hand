# Day17 P3-S2 性能对比数据 v1.0

- 日期：2026-02-24
- 口径：按代码路径统计数据库交互次数（SQL round-trip）
- 说明：本数据用于验证“批量写优化”收益，重点看 DB 往返次数下降

---

## 1. 对比场景 A：Outbox 发布结果回写

### 场景定义
- 单次调度拉取 `N` 条待发送 outbox
- 发送后需要更新状态（SENT 或 FAIL）

### SQL 交互次数对比

| N | 改造前 | 改造后 | 降幅 |
|---:|---:|---:|---:|
| 10 | `1 + 10` | `1 + 2` | -72.7% |
| 50 | `1 + 50` | `1 + 2` | -92.2% |
| 200 | `1 + 200` | `1 + 2` | -98.5% |

说明：
- `1` 为 `listPending(limit)` 查询
- 改造前每条各 1 次 `markSent/markFail`
- 改造后固定 2 次：`markSentBatch` + `markFailBatch`（空集合时可降为 1 次）

---

## 2. 对比场景 B：ShipReminder 卡死任务回收

### 场景定义
- 单次回收拉取 `N` 条 stale RUNNING 任务
- 根据重试次数计算延时档位（2/5/15/30 分钟）

### SQL 交互次数对比（按 4 档位上限）

| N | 改造前 | 改造后（最差） | 降幅 |
|---:|---:|---:|---:|
| 10 | `1 + 10` | `1 + 4` | -45.5% |
| 50 | `1 + 50` | `1 + 4` | -88.2% |
| 200 | `1 + 200` | `1 + 4` | -97.5% |

说明：
- `1` 为 `listStaleRunning(limit)` 查询
- 改造前逐条 `markFail`
- 改造后按“延时档位”分组批量更新，理论上限 4 次

---

## 3. DoD 对齐结论

1. 典型批量场景 DB 交互次数显著下降：已满足。  
2. 批量失败回滚策略：
   - Outbox：`flushPublishResult` 单事务回滚，行为可预期。
   - ShipReminder：分组批量单 SQL，失败可重试，行为可预期。
