# Day17 Step P2-S3：XML 复杂 SQL 规范化说明

- 文档版本：v1.0
- 日期：2026-02-24
- 目标：让保留在 XML 的复杂 SQL 具备统一可读性与可审计性。

---

## 1. 适用范围
以下 SQL 必须按本规范注释：
1. 多表 Join 查询或更新
2. 全文检索、复杂排序
3. 聚合统计（`GROUP BY`、多 CASE 聚合）
4. 锁语义 SQL（`FOR UPDATE`）
5. 状态机条件更新（`id + current_status` 等）

---

## 2. 强制注释模板
每条复杂 SQL 前必须使用以下注释块：

```xml
<!--
  SQL_ID: P2S3-XXX-001
  用途: ...
  输入: ...
  索引依赖: ...
  不迁移原因: ...
-->
```

字段要求：
1. `SQL_ID`：唯一且稳定，不允许重复。
2. `用途`：一句话说明业务目标。
3. `输入`：列出核心参数（不是全部参数）。
4. `索引依赖`：列出关键索引或主键依赖。
5. `不迁移原因`：必须可审计，禁止写“历史原因”这类空话。

---

## 3. 命名规则
1. Product 域：`P2S3-PROD-xxx`
2. Order 域：`P2S3-ORD-xxx`
3. Wallet 域：`P2S3-WAL-xxx`
4. 其他域按同样规则扩展：`P2S3-{域缩写}-xxx`

---

## 4. 变更流程
1. 新增复杂 SQL：先分配 `SQL_ID`，再写 SQL。
2. 修改复杂 SQL：必须同步更新注释与清单文档。
3. 下线复杂 SQL：从清单移除并在变更说明中标记下线原因。

---

## 5. 与 MP 的边界
1. 通用 CRUD / 简单条件查询优先 MP（`BaseMapper + Wrapper`）。
2. 本文档定义的复杂 SQL 可继续保留 XML。
3. 保留 XML 不等于放任不管，必须满足“可读 + 可审计 + 可解释”。
