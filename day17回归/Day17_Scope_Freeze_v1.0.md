## 3. 混合 DAO 策略冻结（核心原则）
### 3.1 选型边界
1. **MP 适用场景（默认）**
   - 单表 CRUD
   - 简单条件查询（等值/范围/排序/分页）
   - 常规列表页、管理后台基础检索
2. **传统 MyBatis XML 保留场景（必须）**
   - 多表复杂联查（复杂 Join、子查询、窗口函数）
   - 强 SQL 优化场景（需精确控制执行计划）
   - 报表/统计/聚合型查询
   - 历史稳定 SQL（改造收益明显低于风险）

### 3.2 分页策略
1. 目标态：新改造模块统一使用 MP `Page<T>`。
2. 过渡态：老模块允许继续使用 `PageHelper`，但**同一接口链路只允许一种分页机制**。
3. 对外响应统一：`PageResult<T>`（字段口径统一）。

### 3.3 简历展示口径（工程价值）
1. 展示 MP 工程化能力：通用 CRUD、自动填充、插件配置、批量能力。
2. 展示传统 MyBatis 深度能力：复杂 SQL 建模、索引/执行计划调优、性能治理。
3. 展示架构判断力：不是“技术替换”，而是“场景化选型 + 边界治理”。

## 5. Phase 1：MP 基建统一（冻结）
### Step P1-S1：统一 MyBatis-Plus 核心配置

  2. 明确 `mapper-locations`、驼峰映射、日志级别口径。
  3. 明确“分页插件单链路不混用”规则。
- 输出：配置类、配置项说明、项目级约束说明。
- DoD：
  1. 应用可正常启动。
  2. 新增 MP 分页接口可运行。
  3. 配置口径在文档中可被他人直接复现。

### Step P1-S2：全局字段与自动填充能力落地
- 目标：统一 `create_time` / `update_time` 写入规则，去掉重复手工赋值。
- 范围：
  1. 定义 `MetaObjectHandler` 自动填充策略。
  2. 统一实体字段命名与注解口径（必要时抽取基类字段）。
  3. 插入/更新链路补充约束与示例。
- 输出：自动填充实现 + 实体规范。
- DoD：
  1. 插入自动写 `create_time/update_time`。
  2. 更新自动刷新 `update_time`。
  3. 主要实体不再重复手工设置时间字段。

### Step P1-S3：通用 Mapper/Service 工程模板固化
- 目标：把可复用 CRUD 代码沉淀成团队模板。
- 范围：
  1. 明确哪些模块继承 `ServiceImpl`。
  2. 统一 Service 层命名和方法语义（`save/update/remove/get/page`）。
  3. 约束何时允许绕过通用方法写自定义 SQL。
- 输出：模板示例（以现有 MP 模块为蓝本）+ 规范清单。
- DoD：
  1. 至少 1 个模块可作为标准样板。

## 6. Phase 2：试点迁移与混合策略落地（冻结）
### Step P2-S1：确定迁移试点模块
- 目标：以低风险模块验证改造方法论，而非全量盲改。
- 范围：
  1. 选 1~2 个业务模块作为首批迁移对象。
  2. 评估每个模块的 SQL 复杂度与回归成本。
  3. 形成“可迁移 / 不迁移（保留 XML）”清单。
- 输出：试点模块清单 + 迁移优先级表。
- DoD：
  1. 每个试点模块都有迁移范围说明。
  2. 每个保留 XML 的接口都有保留理由。

### Step P2-S2：通用 CRUD 迁移到 MP
- 目标：减少重复 SQL 与样板代码，提升可维护性。
- 范围：
  1. Mapper 继承 `BaseMapper<T>`。
  2. Service 继承 `ServiceImpl<M, T>`。
  3. 简单条件查询改为 `QueryWrapper/LambdaQueryWrapper`。
- 输出：迁移后的模块代码与对照说明。
- DoD：
  1. 迁移接口行为与原接口一致（功能无回归）。
  2. Service/Mapper 重复代码显著下降。

### Step P2-S3：复杂 SQL 保留并标准化
- 目标：保留传统 MyBatis 能力，同时提升可读性与可审计性。
- 范围：
  1. 复杂 SQL 继续保留在 XML。
  2. 每条复杂 SQL 增加“用途/输入/索引依赖”说明。
  3. 对高频复杂 SQL 输出执行计划结论。
- 输出：复杂 SQL 清单 + XML 规范化说明。
- DoD：
  1. 所有保留 SQL 都能说明“不迁移原因”。
  2. 关键复杂 SQL 至少完成一次 `EXPLAIN` 校验。

### Step P2-S4：冗余代码清理
- 目标：消除迁移后并存冗余，避免双实现长期共存。
- 范围：
  1. 删除重复 DAO 方法与废弃查询分支。
  2. 收敛重复 DTO/转换逻辑。
  3. 保留兼容层时必须标记淘汰计划。
- 输出：清理后的代码与兼容说明。
- DoD：
  1. 无“新实现 + 旧实现”长期重复调用路径。
  2. 迁移后的代码结构比迁移前更清晰。

## 7. Phase 3：性能优化与 SQL 治理（冻结）
### Step P3-S1：分页全覆盖与大查询收口
- 目标：所有列表查询都可控，杜绝无边界全表扫描。
- 范围：
  1. 列表接口统一接入分页参数（`page`、`pageSize`）。
  2. 统一返回 `PageResult`。
  3. 治理高风险查询（无 where、无 limit、深分页）。
- 输出：分页接口改造清单。
- DoD：
  1. 列表接口 100% 支持分页。
  2. 无默认全量查询接口暴露给业务层。

### Step P3-S2：批量写优化
- 目标：降低数据库往返次数，提升批处理效率。
- 范围：
  1. 能用 MP 批量能力的接口优先改造（`saveBatch/updateBatchById`）。
  2. 对复杂批量场景保留 XML 批量 SQL。
  3. 批量写入补充事务边界与失败回滚策略。
- 输出：批量接口改造记录 + 性能对比数据。
- DoD：
  1. 典型批量场景数据库交互次数下降。
  2. 批量失败可回滚且行为可预期。

### Step P3-S3：慢 SQL 与索引治理
- 目标：将性能优化从“经验驱动”变成“数据驱动”。
- 范围：
  1. 拉取慢 SQL 样本并分级（高/中/低风险）。
  2. 针对高频条件字段建立复合索引（如 `user_id`、`product_id`、`status`、`create_time`）。
  3. 每条优化 SQL 给出 `EXPLAIN` 结论。
- 输出：索引脚本 + SQL 优化清单。
- DoD：
  1. 关键查询无明显全表扫描风险。
  2. 优化前后数据可量化（耗时/扫描行数/调用次数）。

### Step P3-S4：N+1 查询治理
- 目标：减少循环内多次访问数据库导致的线性放大。
- 范围：
  1. 排查 Service 层循环查询。
  2. 优先改为批量查询 + 内存映射组装。
  3. 对保留场景给出业务合理性说明。
- 输出：N+1 治理记录。
- DoD：
  1. 高频接口不再出现明显 N+1 模式。
  2. 改造后接口响应时间有稳定改善。

## 8. Phase 4：事务与幂等闭环（冻结）
### Step P4-S1：事务边界图谱化
- 目标：明确每条关键链路的事务边界，减少隐式不一致。
- 范围：
  1. 梳理跨表写入链路（订单、商品、积分、违规、消息等）。
  2. 为关键业务方法补齐 `@Transactional`。
  3. 定义只读查询与写事务隔离口径。
- 输出：事务边界清单 + 关键方法标注结果。
- DoD：
  1. 关键跨表操作全部有明确事务保护。
  2. 异常路径可验证回滚行为。

### Step P4-S2：幂等机制标准化
- 目标：重复请求不产生重复副作用。
- 范围：
  1. 确定业务幂等键（订单号、业务流水号、事件号等）。
  2. 以唯一索引/唯一约束实现数据库层防重。
  3. 统一重复请求返回语义（成功但已处理/幂等命中）。
- 输出：幂等策略清单 + 唯一约束脚本。
- DoD：
  1. 重复提交核心接口不会产生重复数据。
  2. 幂等命中日志可追踪。

### Step P4-S3：并发更新安全
- 目标：避免并发覆盖和状态错乱。
- 范围：
  1. 更新操作采用条件更新（主键 + 期望状态/版本号）。
  2. 统一处理“影响行数=0”的分支。
  3. 关键状态流转补充并发回归用例。
- 输出：并发安全改造清单。
- DoD：
  1. 关键状态流转具备并发防护。
  2. 并发失败分支返回明确业务语义。

### Step P4-S4：与异步链路一致性对齐
- 目标：同步主流程与异步通知链路一致，不出现“主成功、异步丢失”。
- 范围：
  1. 复用既有 Outbox 思路确保事件可靠投递。
  2. 消费侧幂等去重保持不变并补充说明。
  3. 异步失败不影响主事务成功但可重试补偿。
- 输出：一致性说明与失败恢复流程。
- DoD：
  1. 主链路成功后事件可追踪。
  2. 消费重复不产生重复副作用。

## 10. 分页与返回契约冻结（Day17）
### 10.1 请求参数口径
1. `page`：页码（从 1 开始）
2. `pageSize`：每页条数（统一上限，例如 100）

### 10.2 响应口径
统一 `PageResult<T>`，最小字段集：
- `records`
- `total`
- `page`
- `pageSize`

### 10.3 兼容策略
1. 历史接口返回结构若不一致，允许短期兼容。
2. 兼容期结束后统一切换到 `PageResult<T>`。
3. 每个兼容接口必须在迁移清单中标记下线计划。

## 11. 数据库索引优化冻结（最小集）
> 以下为 Day17 的“最低索引治理要求”，实际以业务 SQL 分析结果为准。

1. 高频按用户查询：`(user_id, create_time)` 复合索引
2. 高频按商品查询：`(product_id, create_time)` 复合索引
3. 状态筛选列表：`(status, create_time)` 复合索引
4. 订单幂等键：唯一索引（如 `order_no`）
5. 业务防重键：唯一索引（如请求幂等键/业务流水号）

验收要求：
1. 每条新增索引都要有对应 SQL 场景。
2. 每条新增索引都要评估写入成本与收益。

## 12. 风险与回滚策略冻结
### 12.1 主要风险
1. 分页机制并存导致行为不一致。
2. 迁移后 SQL 语义偏差导致结果集变化。
3. 事务边界扩大引发锁竞争与性能抖动。
4. 索引新增不当导致写性能下降。

### 12.2 回滚策略
1. 采用“模块级开关/分批发布”而非全量一次切换。
2. 关键接口保留短期回退路径（旧 SQL 或旧实现）。
3. DDL 变更先灰度验证，失败时按变更脚本回滚。
4. 回归失败立即停止下一模块迁移，先清缺陷。
