# 前端短期计划

## 当前进展

目前管理端前端已经完成了以下基础能力：

- 登录页与登录拦截已经接入，可以进入管理端查看页面。
- 管理端首页已经完成基础展示，并能承接后续真实数据接入。
- 用户与商家管理页已经完成列表、分页、筛选、建档、封禁、解封等主流程。
- 用户封禁理由已经补齐前端交互，可以继续做联调与验收。
- 商品审核页已经补齐基础骨架，并可从菜单正常进入。
- 系统设置页已经补齐占位结构，未完成菜单收口已有初步落地。
- 纠纷与违规页已经完成前端页面，并补齐了后端聚合接口接入能力。

当前阶段的重点已经从“把核心流程跑通”切换到“补齐剩余页面、统一交互体验、提高联调完成度”。

---

## 短期目标

短期建议以 1 到 2 周为周期，优先完成以下事项：

### 1. 完成管理端剩余核心页面骨架

优先把左侧菜单里还不能真正落地使用的页面补出来，至少先做到：

- 商品审核页（已完成基础骨架）
- 纠纷与违规页（已完成页面与聚合接口接入）
- 系统设置页（已完成占位版）

这一阶段的目标不是一次把所有细节做满，而是先完成：

- 页面路由可访问
- 基础布局统一
- 空状态、加载态、错误态完整
- 主要操作区和筛选区先搭好

### 2. 把首页和用户页的“假数据区域”继续替换为真实接口

当前首页仍有部分模块偏展示型，后续需要继续核对后端接口并逐步替换：

- 首页指标卡
- 审核队列
- 风控预警
- 纠纷处理区块

如果后端暂时没有聚合接口，前端要先统一成“可切换 mock / real”的结构，避免后面重复改。

### 3. 统一后台弹窗、表单、表格交互规范

现在已经有“人工建档弹窗”和“封禁理由弹窗”这类交互，短期内建议统一一套页面规范：

- 弹窗标题样式统一
- 表单项间距统一
- 主次按钮样式统一
- 提交中、成功后刷新、失败提示方式统一
- 表格空状态文案统一

这样后续新页面越多，前端越不会越写越散。

### 4. 完成未开放菜单的可用性收口

当前有些菜单如果还没完成，不要让用户点进去后没有反馈。短期内建议统一处理为两种方式之一：

- 已完成：正常进入页面
- 未完成：显示“建设中”占位页或禁用入口

这样能避免联调和演示时出现“看起来能点，其实什么都没有”的割裂感。

### 5. 做一轮前端联调验收清单

在剩余关键页面补到可访问后，建议整理一份前端联调验收清单，至少覆盖：

- 登录
- 首页数据加载
- 用户列表分页筛选
- 人工建档
- 封禁 / 解封
- 封禁理由提交
- 导航跳转
- 失效 token 回登录页

这一步很重要，它能帮助你快速判断当前管理端是否已经达到“可以演示、可以继续扩展”的状态。

---

## 建议执行顺序

建议按下面顺序推进，节奏会比较稳：

1. 先统一弹窗、表格、空状态样式
2. 再继续把首页剩余 mock 数据替换成真实接口
3. 再补“退出登录”等登录态收口能力
4. 然后继续做纠纷与违规页的处理动作联调
5. 最后做一轮联调整体验收

---

## 交给 Gemini 的工作方向

后续如果继续让 Gemini 参与前端开发，建议优先让它处理以下类型任务：

- 补页面骨架和路由接入
- 按现有风格补弹窗和表单交互
- 把页面上的 mock 区域替换为真实接口
- 统一页面级的空状态、加载态、错误提示

不建议一开始就让它大改整体视觉体系或重构全部布局，否则很容易把现在已经形成的后台风格打散。

---

## 这份计划的使用方式

这份文档可以作为接下来一小段时间的前端推进说明，后续建议每完成一项就顺手更新：

- 已完成什么
- 当前阻塞点是什么
- 下一个页面准备做什么

这样你后面无论是自己继续写，还是交给 Gemini，都能接得更顺。
