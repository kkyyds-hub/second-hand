# Day18 范围冻结：平台一致性与安全收口

- 项目：二手交易平台（`demo-platform` / `com.demo`）  
- 模块：Day18 一致性与安全收口（事务、幂等、异步一致性、安全、防作弊、性能、测试）  
- 文档版本：v1.0（首次冻结）  
- 冻结日期：2026-02-24  
- 文档性质：**里程碑冻结文档（允许跨多天执行）**  
- 目标：在现有 Day17 基础上，完成“可审计、可回归、可移交”的一致性与安全收口。

---

## 0. 范围定位（冻结）

1. Day18 是“收口阶段”，核心目标是把已有能力标准化并补齐关键缺口。  
2. Day18 不以新增大业务功能为目标，重点是稳定性、安全性、可观测性。  
3. 每个 Step 必须有可验收产物（代码/脚本/文档/回归结果）。  
4. 高风险项一律要求“证据化交付”（配置、日志、SQL、测试记录可追溯）。

---

## 1. 边界决策（已冻结）

1. 继续采用“本地事务 + Outbox + 补偿任务”一致性主方案：**YES**  
2. 引入 Seata/Saga/TCC 分布式事务框架：**NO（Day18 不引入新框架）**  
3. 核心接口幂等键 + 数据库唯一约束双保险：**YES**  
4. MQ 维持至少一次投递 + 消费端去重：**YES**  
5. 缓存一致性策略文档化并在关键链路落地：**YES**  
6. 数据库读写分离改造：**NO（Day18 不做架构级拆分）**  
7. 统一乐观并发控制（CAS/条件更新）于关键状态迁移：**YES**  
8. 密码/令牌等敏感数据加固（加密、脱敏、泄露面收口）：**YES**  
9. SQL 注入防护基线（参数化 + 白名单 + MP 防全表写）：**YES**  
10. XSS 防护统一策略（输入约束 + 输出规范）：**YES**  
11. CSRF Token 全量改造：**NO（当前 JWT Header 架构，Day18 产出风险说明与替代策略）**  
12. 安全审计日志口径统一（来源、动作、对象、结果）：**YES**  
13. ELK/Kibana 实际部署：**NO（Day18 输出接入规范与告警字段）**  
14. 风控能力按“登录失败冻结 + 信用评分 + 异常交易规则”收口：**YES**  
15. 限流/削峰策略在高风险入口形成明确方案：**YES**  
16. 回归测试与安全测试纳入冻结门槛：**YES**
